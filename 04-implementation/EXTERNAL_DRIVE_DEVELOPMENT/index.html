    External Drive Development Guide¶

    ‚ö†Ô∏è Important: Developing on external drives (macOS /Volumes/) requires special considerations

    This guide addresses the unique challenges of developing the ToolBoxAI platform on external drives and provides tested solutions.

    üéØ Quick Summary¶

    Method Pros Cons Recommended
    Docker Full compatibility, no issues Requires Docker ‚úÖ YES
    Native Direct file access Requires workarounds ‚ö†Ô∏è Conditional

    Recommendation: Use Docker for all development on external drives.

    üö® The Problem¶

    System Error -88¶

    Native binaries fail to execute on external drives with:

    Error: spawn Unknown system error -88

    Affected Tools: - esbuild (used by Vite for transpilation) - rollup (used by Vite for bundling) - Other native Node.js addons

    Root Cause: macOS external drive filesystem limitations prevent execution of native binaries.

    ‚úÖ Solution 1: Docker (Recommended)¶

    Why Docker?¶

    • Full Linux filesystem - No macOS external drive limitations
    • Consistent environment - Same for all developers
    • Zero workarounds - Everything just works
    • Production parity - Matches deployment environment

    Setup Instructions¶

    # Navigate to project
cd /Volumes/G-DRIVE/Development/ToolBoxAI-Solutions

# Start all services
docker compose -f infrastructure/docker/compose/docker-compose.yml \
               -f infrastructure/docker/compose/docker-compose.dev.yml up -d

# Verify services running
docker compose ps

# View logs
docker compose logs -f

# Access services
open http://localhost:5179  # Dashboard
open http://localhost:8009/docs  # API

    Development Workflow¶

    # Make code changes directly on external drive
# Files are mounted into containers via volumes

# Hot reload works automatically for:
# - Frontend (Vite HMR)
# - Backend (uvicorn --reload)

# Restart specific service if needed
docker compose restart dashboard-frontend

# Shell into container for debugging
docker compose exec dashboard-frontend /bin/sh
docker compose exec fastapi-backend /bin/bash

    Benefits¶

    ‚úÖ No native binary issues ‚úÖ No NPM workarounds needed ‚úÖ Full Linux filesystem support ‚úÖ Production-ready environment ‚úÖ Consistent across all team members ‚úÖ Hot reload works perfectly

    ‚ö†Ô∏è Solution 2: Native with Workarounds¶

    When to Use¶

    Only use native development if: - Docker is not available - You need direct Node.js debugging - You're testing platform-specific issues

    Required Workarounds¶

    1. Vite Configuration¶

    Convert vite.config.ts to vite.config.js to avoid esbuild:

    cd apps/dashboard

# Rename TypeScript config to JavaScript
mv vite.config.ts vite.config.js

    Why? TypeScript requires esbuild for transpilation, which fails on external drives.

    2. NPM Installation¶

    Always use --no-bin-links flag:

    # Initial install
npm install --no-bin-links --legacy-peer-deps

# Adding new packages
npm install --no-bin-links --legacy-peer-deps <package>

# Clean install
rm -rf node_modules package-lock.json
npm install --no-bin-links --legacy-peer-deps

    Why? Prevents NPM from creating symlinks to native binaries.

    3. Package Scripts¶

    Update package.json for external drive compatibility:

    {
  "scripts": {
    "install:external": "npm install --no-bin-links --legacy-peer-deps",
    "install:clean": "rm -rf node_modules package-lock.json && npm install --no-bin-links --legacy-peer-deps",
    "dev": "vite",
    "build": "vite build",
    "preview": "vite preview"
  }
}

    Development Workflow¶

    # Backend (Python - works fine)
cd apps/backend
python3 -m venv venv
source venv/bin/activate
pip install -r requirements.txt
uvicorn main:app --reload

# Frontend (Node.js - requires workarounds)
cd apps/dashboard
npm run install:external  # Use custom script
npm run dev

    Limitations¶

    ‚ùå Cannot use vite.config.ts (TypeScript config) ‚ùå Must use --no-bin-links for every install ‚ùå Some dev tools may not work ‚ùå Type checking requires separate command ‚ùå Build may be slower without esbuild

    üîß Troubleshooting¶

    Issue: esbuild Error¶

    Error: esbuild cannot run on this file system
spawn Unknown system error -88

    Solution: Use Docker or convert vite.config.ts to .js:

    # Option 1: Use Docker
docker compose up -d dashboard-frontend

# Option 2: Convert to JavaScript
mv vite.config.ts vite.config.js

    Issue: Rollup Error¶

    Error: Cannot find module @rollup/rollup-linux-x64-gnu

    Solution: Use Docker or install with --no-bin-links:

    # Option 1: Use Docker
docker compose up -d

# Option 2: Reinstall with flag
npm run install:clean

    Issue: Permission Denied¶

    Error: EACCES: permission denied

    Solution: Check file permissions or use Docker:

    # Fix permissions (if needed)
chmod -R u+w /Volumes/G-DRIVE/Development/ToolBoxAI-Solutions

# Or use Docker (no permission issues)
docker compose up -d

    Issue: Slow File Watching¶

    External drives can be slower for file watching:

    Solution: Configure Vite polling:

    // vite.config.js
export default defineConfig({
  server: {
    watch: {
      usePolling: true,  // Enable polling for external drives
      interval: 1000     // Check every 1 second
    }
  }
});

    üéØ Best Practices¶

    For External Drive Development¶

    1. Always use Docker for primary development
    2. Keep venv in project (not on system Python)
    3. Use polling for file watching if needed
    4. Regular backups (external drives can fail)
    5. Git commit often (don't lose work)

    Docker-Specific Tips¶

    # Prune unused resources regularly
docker system prune -a --volumes

# Check Docker disk usage
docker system df

# Limit Docker resource usage (if needed)
# Edit: ~/.docker/daemon.json
{
  "storage-driver": "overlay2",
  "storage-opts": ["size=50G"]
}

    Backup Strategy¶

    # Backup entire project
rsync -av --progress \
  /Volumes/G-DRIVE/Development/ToolBoxAI-Solutions \
  ~/Backups/ToolBoxAI-$(date +%Y%m%d)

# Backup Docker volumes
docker compose down
tar -czf volumes-backup.tar.gz ./data

# Verify git status before backup
git status
git log --oneline -5

    üîç Verification Checklist¶

    Before Starting Development¶

    • External drive is properly mounted
    • Docker Desktop is running
    • Docker Compose v2 is available (docker compose version)
    • At least 10GB free space on external drive
    • Git is configured correctly
    • Project is cloned to external drive

    After Setup¶

    • docker compose ps shows all services running
    • Dashboard accessible at http://localhost:5179
    • API docs accessible at http://localhost:8009/docs
    • Hot reload works when editing files
    • Type checking passes (npm run typecheck)
    • Tests pass (npm test, pytest)

    üÜò Getting Help¶

    If Docker Doesn't Work¶

    1. Check Docker Desktop is running
    2. Verify Docker Compose v2: docker compose version
    3. Check logs: docker compose logs
    4. Try rebuild: docker compose up --build
    5. Ask in #development channel

    If Native Development Fails¶

    1. Verify using --no-bin-links flag
    2. Check vite.config.js (not .ts)
    3. Try clean install: npm run install:clean
    4. Consider switching to Docker
    5. Check Troubleshooting section above

    üí° Pro Tips¶

    Speed Optimizations¶

    # Use BuildKit for faster Docker builds
export DOCKER_BUILDKIT=1

# Parallel NPM installs
npm install --no-bin-links --legacy-peer-deps --maxsockets=15

# Use Docker layer caching
docker compose build --no-cache  # Only when needed
